# integrate-stage

The repo evolves one stage at a time. Each new stage arrives as a single Python file plus a set of sample PNGs (produced in a sandbox with a `--save-prefix` CLI that this repo doesn't use). The integration work is exactly the same every time — only the new features differ. This skill captures the mechanical part so each new stage is one well-shaped commit, and the JOURNAL entry that documents the stage is its own separate, well-shaped commit later.

The skill has two phases that fire on different prompts:

- **Phase A — Integration.** User drops the file paths (sometimes with a zip of sample outputs). Output: new `logical_surface_model_v{N}.py` in repo root, populated `outputs/v{N}/runs/example/`, updated `README.md`, one commit.
- **Phase B — JOURNAL entry.** User asks for "Stage {N} in JOURNAL.md" *after* integration has landed. Output: appended `## Stage {N}` section in `JOURNAL.md`, one commit.

Never run Phase A and Phase B in the same turn unless the user explicitly asks for both. The pattern from v3/v4/v5 is: integrate → user reviews the actual outputs → user asks for JOURNAL later. Drafting the JOURNAL before the user has seen the run wastes work — they may want different framing once they see what the numbers actually look like.

## Phase A — Integrate a new stage

### A.1 — Detect inputs

The user's message will name one or both of:

- A new model script: `logical_surface_model_v{N}.py` (typically in `~/Downloads`).
- A sample outputs zip: `logical_surface_v{N}_outputs.zip` (typically with `sample_output.txt` + 6–8 PNGs named `sample_v{N}_*.png`).

Parse `{N}` from the filename. If the version is ambiguous (e.g., the file is named `logical_surface_model.py` with no version), use `AskUserQuestion` to confirm.

The outputs zip is **reference material** for the user's expected results — do not unzip it into the repo. The repo's own `outputs/v{N}/runs/example/` is generated by running the integrated script, not copied from the zip. The zip's `sample_output.txt` is useful as a sanity check against the integrated run's stdout (events table, relations table, metrics block).

### A.2 — Copy + refactor

Copy the file unchanged: `cp ~/Downloads/logical_surface_model_v{N}.py logical_surface_model_v{N}.py` in the repo root.

Then apply the standard refactor. The reference implementations are `logical_surface_model_v3.py`, `logical_surface_model_v4.py`, and `logical_surface_model_v5.py` — read the most recent one's `# Analysis entry point` section as the template. The refactor has five mechanical parts:

1. **Imports + module constants.** At the top of the file, add `hashlib`, `json`, `uuid` to the standard imports, and add `from datetime import datetime, timezone`. If the new version adds dataclasses that need serialization to JSON (claims, claim relations, etc.), also add `asdict` to the `dataclasses` import. Then add two module-level constants right after the imports:

   ```python
   VERSION_TAG = "v{N}"
   DEFAULT_OUTPUT_ROOT = Path(__file__).resolve().parent / "outputs"
   ```

   Also rewrite the long header comment block so the "Compared with v{N-1}, this version adds…" list reflects what's actually new in v{N}, not what was new in v3. The "Run:" examples should show `--no-plots` instead of `--save-prefix`.

2. **Run-folder helpers.** Inside the `# Analysis entry point` section, immediately before `def analyze_text(...)`, insert these five helpers verbatim from the most recent integrated version: `operator_catalog_signature`, `generate_run_id`, `prepare_run_directory`, `write_run_params`, `write_run_metrics`. The `write_run_params` payload must include `intensifiers`, `hedges`, and any other module-level vocabularies the new version uses (v4 added `MENTION_VERBS`/`MENTION_NOUNS`/`TEXTUAL_SUBJECTS`; v5 kept those four). Check the file for new `*_VERBS` / `*_NOUNS` / `*_PHRASES` / `*_WORDS` module constants and include the relevant ones — the rule is "anything that controls model behavior should be recorded in params so a future run is reproducible." Skip purely internal sets that are used in one function and never tuned.

3. **`write_run_analysis` (only if new structured outputs exist).** If the new version produces structured outputs that don't fit a flat metrics dict (v5's `claims` list, `claim_relations` list, inferred `genre`, `diagnostics` list), add a `write_run_analysis` helper that serializes them to `analysis.json` next to `metrics.json`. Use `asdict` for frozen dataclasses. The v5 implementation is the template:

   ```python
   def write_run_analysis(run_dir, *, graph, genre, genre_confidence, genre_evidence, diagnostics):
       payload = {
           "genre": {"name": genre, "confidence": genre_confidence, "evidence": genre_evidence},
           "diagnostics": list(diagnostics),
           "claims": [asdict(c) for c in graph.claims],
           "claim_relations": [asdict(r) for r in graph.claim_relations],
       }
       (run_dir / "analysis.json").write_text(json.dumps(payload, indent=2), encoding="utf-8")
   ```

   For a v{N} with different structured outputs, adjust the payload shape. The principle: flat numerical metrics → `metrics.json`; structured records (lists of objects, classification verdicts, free-text diagnostics) → `analysis.json`.

4. **`analyze_text` signature + body.** Replace the sandbox-style signature:

   ```python
   def analyze_text(text, *, show_plots=True, save_prefix=None, x_resolution=280,
                   y_resolution=140, max_relation_to_operator_ratio=1.25):
   ```

   with the repo's signature:

   ```python
   def analyze_text(text, *, show_plots=True, save=True, output_root=None, run_id=None,
                   x_resolution=280, y_resolution=140,
                   operator_cross_family_width=0.55, relation_cross_family_width=0.42,
                   max_relation_to_operator_ratio=1.25):
   ```

   In the body, when `save` is `True`: resolve `run_dir = prepare_run_directory(root, run_id_or_generated)`, call `write_run_params(...)`, `write_run_metrics(...)`, and `write_run_analysis(...)` if applicable. The plot save paths become `run_dir / "total_surface.png"`, `run_dir / "operator_heatmap.png"`, etc. — drop the `_v{N}_` filename prefix the sandbox version used. Return `run_id` and `run_dir` in the result dict.

5. **`parse_args` + `__main__`.** Replace the `--save-prefix` argument with the standard set: `--no-save`, `--output-dir`, `--run-id`, plus `--operator-cross-family-width` and `--relation-cross-family-width` (both default to 0.55 and 0.42 respectively). Pass them through to `analyze_text` in `__main__`.

The PNG filenames in the run directory follow this convention (always lowercase, no version prefix, descriptive only):

- v1 only — `surface.png`, `heatmap.png`
- v2 — `total_surface.png`, `total_heatmap.png`, `operator_heatmap.png`, `relation_heatmap.png`
- v3+ — splits relation into `relation_raw_heatmap.png` + `relation_normalized_heatmap.png`
- v4+ — adds `relation_graph.png`
- v5+ — adds `claim_graph.png` and `vector_field.png`

When v{N} introduces a new plot, use a name in the same `<noun>_<noun>.png` style — and update the README's "PNG filenames per version" list to add the new entry.

### A.3 — Verify the build_surface signature

Before changing the CLI, grep for `build_surface(` in the new file. If the function still accepts `operator_cross_family_width` and `relation_cross_family_width` parameters (it should — v3/v4/v5 all do), pass them through from `analyze_text`. If a future version drops them, drop the CLI flags too and note it in the commit message.

### A.4 — Run the script

```bash
uv run logical_surface_model_v{N}.py --no-plots --run-id example
```

`--run-id example` makes the output folder deterministic at `outputs/v{N}/runs/example/` — that's the well-known location the README points to. `--no-plots` skips opening matplotlib windows but still writes PNGs (the run folder gets the same artifacts as a normal run, just without the GUI).

Verify the folder contains: `params.json`, `input_text.txt`, `metrics.json`, `analysis.json` (if the new stage uses it), and one PNG per plot the new version produces. If any are missing, the refactor was incomplete — debug before continuing.

Cross-check the printed metrics against the user's expected numbers (often in the chat message that delivered the file). Mismatches mean either the refactor changed something inadvertently (most likely cause: dropped or renamed a kwarg) or the sandbox-version run was on a different sample text. Either way, surface the mismatch to the user before committing.

### A.5 — Update README

Five targeted edits, all in `README.md`:

1. **Install/run section** — add `uv run logical_surface_model_v{N}.py` with a one-line description of what the version adds.
2. **CLI flags table** — change "All four versions share the same core flags" → "All five versions" (etc.); update the `--x-resolution` / `--y-resolution` defaults footnote to mention v{N}.
3. **Version-specific knobs** — add v{N} to the row that lists `--operator-cross-family-width` / `--relation-cross-family-width` / `--max-relation-ratio` (v3 onwards share the same knobs).
4. **PNG filenames per version** — add a `- **v{N}** — same as v{N-1} plus <new files>` bullet describing the new PNGs.
5. **"What's in here"** — add `logical_surface_model_v{N}.py    # Stage {N}: <one-line description>` to the file listing.
6. **Python module example** — update the import in the example from `from logical_surface_model_v{N-1}` to `from logical_surface_model_v{N}`. Update the `save=True` comment to point at `outputs/v{N}/runs/{run_id}/`.

### A.6 — Commit

Single commit, message follows the established pattern: `Stage {N}: <one-line description of what the version adds>`. Body should explain the additions in 2–6 paragraphs and end with the canonical-sample headline numbers so the commit message is itself a mini status snapshot. End with the standard `Co-Authored-By: Claude` trailer.

Stage the v{N} file, the entire `outputs/v{N}/` directory, and `README.md`. If `JOURNAL.md` is currently unstaged (e.g., the user added a Stage {N-1} entry before this integration), commit it *separately first* with `Stage {N-1}: JOURNAL entry` so the history reads as two clean commits, not one mixed one.

Do not push.

## Phase B — Write Stage {N} in JOURNAL.md

Fires when the user asks for a JOURNAL entry for a specific version that is already integrated. Do not write the JOURNAL entry as part of Phase A — the user typically reviews the actual run first and may want the framing adjusted based on what the numbers turn out to look like.

### B.1 — Read the canonical run

Before writing anything:

1. Read `outputs/v{N}/runs/example/metrics.json` for exact numbers.
2. Read `outputs/v{N}/runs/example/analysis.json` if it exists (for claims, claim relations, genre, diagnostics).
3. Read `outputs/v{N-1}/runs/example/metrics.json` for the prior version's numbers — every Stage entry includes a v{N-1} → v{N} delta on the metrics that changed.
4. Read the relevant new functions in `logical_surface_model_v{N}.py` with file:line references — the JOURNAL is more useful when it grounds each claim in a specific function.
5. Read the previous Stage section in `JOURNAL.md` to match the exact subsection naming and structure.

### B.2 — Structure

Append the section after the last `---` separator at the end of `JOURNAL.md`. The structure has nine fixed parts, in order:

1. **`## Stage {N} — <descriptive title> (`logical_surface_model_v{N}.py`)`** — title line.
2. **`**Status:** current. Entry point: `uv run logical_surface_model_v{N}.py`** — with CLI shape if different from v{N-1}.
3. **`### Goal`** — 2–4 paragraphs naming which prior-stage limitations the new version targets and which it deliberately defers. The first paragraph should be a single concrete sentence: what conceptual move does v{N} make?
4. **`### Model`** — has subsections:
   - **`#### Diff against v{N-1} (only what changed)`** — a numbered table with columns `#` / `Decision` / `v{N-1}` / `v{N}` / `Why`. 8–12 rows is the right range. Each row is one mechanical change, not a feature description.
   - **One subsection per non-trivial new mechanism** — with a `(`file_path:line_number`)` reference in the heading or first sentence. Include the actual formula or pseudocode block where it clarifies; cite the existing code rather than reproducing it verbatim when the existing code is long.
5. **`### Results — same canonical sample text`** — events count, the changed event/claim rows (not all rows — only those that changed from v{N-1}), inferred relations or graph edges, headline metrics in a v{N-1} → v{N} delta table.
6. **`### Interpretation`** — what the numbers mean. Scorecard of which prior-stage limitations were addressed. Explicitly call out metrics that moved in the "wrong" direction and explain *why they aren't regressions* (this happened in v4 with scope_coverage_ratio).
7. **`### Limitations of v{N} (and what they motivate)`** — 5–8 numbered limitations. Each has a root cause traced to a specific file:line where possible, and a `→ **v{N+1} should …**` action item.
8. **`### Concrete v{N+1} entry conditions`** — a fenced block of one-line checklist items that v{N+1} must satisfy to be considered done. These map 1:1 to the limitations above.
9. **`### Open questions to revisit at v{N+1}`** — 3–5 bullets, framed as actual *questions* (not statements), about design decisions v{N} made that v{N+1} should revisit.

End with `---` on its own line, so the next Stage entry has a clean separator to insert before.

### B.3 — Style invariants

- Numbers are exact, copy-pasted from `metrics.json` rounded to 3–4 decimals (e.g., `0.808` not `~0.81`).
- File:line refs use the format `logical_surface_model_v{N}.py:1234` — these are clickable in the user's terminal.
- Never copy the entire diff against v{N-1} into the JOURNAL — the diff table summarizes *what* changed; the sub-mechanism subsections explain *how*; the git commit holds the literal diff. JOURNAL is the narrative layer.
- When a limitation in v{N-1} is *not* fixed in v{N}, say so explicitly with the carry-over notation: "v{N-1} limitation 4 still alive in v{N}".
- Avoid adjectives like "beautiful", "elegant", "great". Use specific verbs ("the metric rose from X to Y because …", "the bug surfaces when …"). The JOURNAL should read as a research log, not marketing.
- Match the prior Stages' tone: terse, direct, technical. Stages 1–5 are the calibration.

### B.4 — Commit

Single commit, message `Stage {N}: JOURNAL entry`. Body is 8–14 lines summarizing what the entry covers. End with the standard `Co-Authored-By: Claude` trailer.

## What this skill does NOT do

- Does not modify `logical_surface_model_v{N-1}.py` and earlier — they are frozen records of their stage. If a bug is discovered in an old version, document it in the relevant Stage's "Limitations" section, do not patch the file.
- Does not run linters / type-checkers / tests — this project has none configured (see `CLAUDE.md`).
- Does not push to a remote — the user pushes when they're ready.
- Does not generate JOURNAL entries proactively during Phase A — only on explicit Phase B request.
- Does not unzip the user's sandbox outputs into the repo — the repo's `outputs/v{N}/runs/example/` is always produced by running the integrated script, never by copying.

## Recovery

If a partial integration is in the working tree from a prior failed run:

- Inspect `git status` first. Untracked `logical_surface_model_v{N}.py` and `outputs/v{N}/` from a half-done attempt can be deleted if the user confirms — do not silently overwrite.
- If the script itself has been edited mid-integration but the run hasn't been generated, finish the refactor before running. Partial refactor + a stale run from `--save-prefix` filenames is the worst case.
- The reference implementations are `logical_surface_model_v3.py`, `logical_surface_model_v4.py`, `logical_surface_model_v5.py`. Read the most recent one when in doubt — that's the canonical template for the run-folder layout.
